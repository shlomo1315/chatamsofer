+# עיצוב: שדרוג פורטל בתי החלמה
+
+תאריך: 2026-07-09
+מצב: מאושר, מוכן ליישום
+
+## רקע
+
+פורטל בתי החלמה (`/admin/maternity/recovery` בצד המשרד, `/portal/maternity/[home]` בצד בית ההחלמה) מנהל מימוש זכאות החלמה של יולדות. המזכירות של בית ההחלמה מזינה סכום שמומש, מספר לילות ומספר קבלה. המסמך מגדיר חבילת שינויים: תיקון באג "כניסה מהירה", שדות חובה + העלאת קובץ קבלה, חלונית אימות לפני שליחה, נעילה + בקשת תיקון, מיילים והתראות, ועיצוב מחדש.
+
+**עיקרון מנחה: פרודקשן חי.** רק רשומות חדשות מהיום עוברות את הזרימה החדשה. רשומות קיימות לא נוגעים בהן. אין migration שמשנה נתונים קיימים. migrations סכמתיים ירוצו ידנית על ידי המשתמש.
+
+## הגדרות מונחים
+
+- **executed** — הערך של `recovery_amount_status` היום כשבית החלמה מזין סכום. פירושו "בוצע, אין צורך באישור נוסף". נשאר כך — **אין** שלב אישור אנושי מצד המשרד.
+- **נעילה** — לאחר שבית החלמה מאשר בחלונית, הרשומה נעולה לעריכה בצד בית החלמה עד שהמשרד פותח אותה מחדש.
+- **y-email** — כתובת המייל שמשמשת את כלל פניות היולדות (`report_email` המוגדר לבית החלמה, ה-y…@gmail.com בצילום). נעשה בו שימוש כיעד למיילי התראה.
+
+## רכיבי השינוי
+
+### א. תיקון "כניסה מהירה"
+
+**בעיה:** `app/api/admin/maternity/portal-login/route.ts:26-31` בודק `.eq('name', home)` מול טבלת `recovery_homes`. בתי החלמה מברירת-מחדל (`DEFAULT_HOMES` ב-`page.tsx:8`) מוזרקים לתצוגה גם בלי שורה בטבלה, ולכן הבדיקה נכשלת וה-route מפנה חזרה בשקט — נראה כאילו הכפתור לא עובד.
+
+**פתרון:**
+1. ב-route: אם בית החלמה קיים ב-`DEFAULT_HOMES` אך אין לו שורה — ליצור שורה (upsert) על-the-fly, או לדלג על בדיקת הקיום עבור שמות מברירת-מחדל. עדיף upsert כדי שהכל יישאר עקבי.
+2. במקרה כשל אמיתי (שם לא מוכר) — להפנות עם פרמטר שגיאה שיוצג כ-toast במסך, במקום redirect שקט.
+
+### ב. שדות חובה + העלאת קובץ קבלה
+
+**שדות חובה בפורטל** (`app/portal/maternity/[home]/PortalApp.tsx`):
+- סכום שמומש (כבר חובה)
+- מספר לילות (**חדש — היום לא חובה**)
+- מספר קבלה (כבר חובה)
+- **קובץ קבלה — חדש, חובה**
+
+**קובץ קבלה:**
+- שדה העלאה (תמונה/PDF) בטופס הפורטל.
+- נשמר ב-Supabase Storage (אותו דפוס כמו `birth_certificate_url` / מסמכים קיימים).
+- URL נשמר בשדה DB חדש `recovery_receipt_url` בטבלת `maternity_aids`.
+- ולידציה בצד לקוח (הכפתור disabled בלי קובץ) ובצד שרת (`/api/portal/recovery-amount/route.ts` דוחה בלי URL קבלה).
+- מוצג בכרטסת הלידה (`app/admin/maternity/[id]/page.tsx`) ליד מספר הקבלה, כמו `DocCard`/`DownloadDocButton` הקיימים.
+
+### ג. חלונית אימות לפני שליחה ("בדוק פעמיים")
+
+**אין** אישור אנושי מצד המשרד. החלונית היא מנגנון מניעת-טעויות בלבד.
+
+- כפתור "סמן כבוצע" → **"אישור ושליחה"**.
+- בלחיצה: ולידציה שכל השדות מלאים, ואז נפתחת חלונית סיכום (Modal קיים, `components/ui/Modal.tsx`).
+- החלונית מציגה את **כל** הפרטים שהוזנו: סכום, לילות, מספר קבלה, שם הקובץ שהועלה.
+- כותרת: *"נא לעבור בעיון על כל הפרטים ולוודא את נכונותם. לאחר האישור לא ניתן יהיה לערוך את הנתונים."*
+- כפתורים: "חזור לתיקון" / "אני מאשר/ת ושולח/ת".
+- באישור → POST ל-`/api/portal/recovery-amount` → הרשומה נכנסת מיידית כ-`executed` **ונעולה**.
+
+### ד. נעילה + בקשת תיקון
+
+**שדה DB חדש:** `recovery_locked` (boolean, default false) בטבלת `maternity_aids`. (migration ידני.)
+
+**נעילה:**
+- באישור החלונית (סעיף ג') — `recovery_locked = true`.
+- בצד בית החלמה: רשומה נעולה מציגה את הנתונים עם תווית "ננעל" ו**אין** כפתור עריכה — במקומו כפתור **"בקש תיקון"**.
+
+**בקשת תיקון:**
+- כפתור "בקש תיקון" בפורטל → POST `/api/portal/recovery-request-edit` (route חדש).
+- מפעיל: (1) מייל ל-y-email, (2) רישום שהרשומה ממתינה לפתיחה (שדה/דגל שהחלונית אצל המשרד קוראת).
+- אינו משנה את הנעילה — רק מודיע.
+
+**פתיחה מחדש (צד משרד):**
+- כפתור **"פתח לעריכה"** ליד הרשומה במסך הראשי ובכרטסת הלידה → POST route שמאפס `recovery_locked = false`.
+- אחרי פתיחה, בית החלמה יכול לערוך שוב; עריכה מחודשת + אישור בחלונית → נועל שוב.
+
+### ה. מיילים והתראות
+
+**מיילים** (דרך `lib/sendMail.ts` → `deliverMail()`, Resend קיים):
+1. בית החלמה שולח נתונים (מימוש זכאות, סעיף ג' באישור) → מייל ל-y-email.
+2. בקשת תיקון (סעיף ד') → מייל ל-y-email.
+
+תבנית מייל חדשה ב-`lib/emailTemplates.ts` בסגנון `shell()` הקיים.
+
+**חלונית התראה בכניסה למערכת** (צד משרד):
+- מנגנון: בכניסה ל-`/admin/maternity/recovery`, טעינת כל האירועים (שליחת נתונים + בקשות תיקון) שקרו מאז הכניסה האחרונה של המשתמש.
+- "כניסה אחרונה" נשמרת per-user (localStorage בצד לקוח מספיק כפתרון ראשוני; אם נדרש cross-device — שדה DB. ברירת מחדל: localStorage).
+- מוצג כ-toast/חלונית תחתונה (מנגנון `Toast`/`NewMailToast` קיים) עם רשימת היולדות שאושרו/שביקשו תיקון.
+
+### ו. אישור והצגה בכרטסת הלידה
+
+`app/admin/maternity/[id]/page.tsx`:
+- מספר קבלה — כבר מוצג (`page.tsx:430`). לוודא הצגה תקינה.

+- כפתור "פתח לעריכה" (סעיף ד') — נגיש גם מכאן.

+### ז. עיצוב מחדש בינוני
+
+**היקף:** פורטל בית החלמה (`PortalApp.tsx` + סגנונות) ומסך "עזר יולדות" הפנימי (`RecoveryHomesView.tsx` / `MaternityTable.tsx`).
+
+**בעיה מרכזית:** גלישת עמודות מחוץ למסך — צריך לגלול צידה כדי לראות. חייב "להיפרס מיד".
+
+**פתרונות:**
+- תיקון ה-layout כך שכל העמודות נכנסות ברוחב המסך בלי גלילה אופקית (בדיקת `min-width`/`white-space`/`table-layout`, וידוא שהכרטיסים הפנימיים לא מרחיבים את הטבלה).
+- ריווח נדיב: padding שורות/תאים, גובה שורה, מרווח בין בלוקים.
+- טיפוגרפיה: גדלים היררכיים ברורים, משקלים.
+- עיצוב מחדש של הבאדג'ים/הכרטיסים הצבעוניים (הבועות הירוקות "היולדת מימשה…", באדג' "תאומים", "ימים") — קונסיסטנטיים ונקיים.
+- עקביות בין שני הפורטלים.
+
+**ללא שינוי מבני עמוק** — נשאר על מבנה הטבלה/נתונים הקיים, משתמשים ב-frontend-design skill לכיוון אסתטי.
