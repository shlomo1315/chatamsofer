## ⚠️ שלב 0 — לפני הכל: משתני סביבה (חובה!)

הוסף/ודא ב-Vercel (או Railway) → Environment Variables **לפני** הפריסה:

| משתנה | ערך | למה זה קריטי |
|-------|------|--------------|
| `OTP_NONCE_SECRET` | מחרוזת אקראית ארוכה (32+ תווים) | הקוד עכשיו **fail-closed**. בלי זה **ההרשמה תישבר** (זריקת שגיאה). אם כבר מוגדר — ודא שהוא קיים. |
| `PORTAL_AUTH_SECRET` | מחרוזת אקראית ארוכה (32+ תווים) | חתימת עוגיית הפורטל. אם חסר — נופל אוטומטית ל-`OTP_NONCE_SECRET`, אז מספיק שאחד מהם קיים. |

> ליצירת מחרוזת אקראית: `openssl rand -base64 32` (או כל generator).

**אם `OTP_NONCE_SECRET` כבר היה מוגדר בפרודקשן** — אין שינוי למשתמשים. אם לא היה — nonces ישנים יפסלו (משתמש באמצע הרשמה יצטרך להתחיל מחדש; זניח).

## שלב 1 — הרצת ה-migrations ב-Supabase

Supabase → SQL Editor. הרץ **כל קובץ בנפרד**, ובדוק את בדיקות השפיות שבתחתית כל קובץ.

### 1a. הידוק RLS
קובץ: `supabase/migrations/20260707_rls_staff_hardening.sql`

מה זה עושה: מחליף `auth.role()='authenticated'` (כל session) ב-`is_staff()` (רק צוות פעיל הרשום ב-profiles), ומוסיף `WITH CHECK` שחסר ב-UPDATE.

**בדיקה מיד אחרי (כמשתמש צוות מחובר):**
```sql
select public.is_staff();          -- אמור: true
select count(*) from beneficiaries; -- אמור: המספר הרגיל
```
אם `is_staff()` מחזיר `false` לצוות אמיתי → ודא שיש לו שורה ב-`profiles` עם `is_active=true`. במקרה חירום: בלוק ה-ROLLBACK בתחתית הקובץ.

### 1b. hash לסיסמאות פורטל
קובץ: `supabase/migrations/20260707_portal_password_hash.sql`

מה זה עושה: מוסיף `password_hash`, מגבב את הסיסמאות הקיימות (bcrypt), מוסיף פונקציות `hash_portal_password` + `verify_portal_password`.

**בדיקה מיד אחרי:**
```sql
select home_name,
       (password_hash is not null) as has_hash,
       (password_hash = crypt(password, password_hash)) as matches
from recovery_portals;
-- has_hash ו-matches אמורים להיות true בכל שורה
```

## שלב 3 — בדיקות עשן אחרי הפריסה

בדוק ידנית שהזרימות הקריטיות עובדות:

- [ ] **לוגין צוות** → כניסה למערכת, רואים נתונים בכל המסכים (RLS לא חוסם).
- [ ] **פורטל בית החלמה** → כניסה עם הסיסמה הקיימת עובדת (hash). *כל המשתמשים יתבקשו להתחבר מחדש פעם אחת* (העוגייה הישנה כבר לא תקפה — זה צפוי).
- [ ] **הרשמה ציבורית** (OTP) → קבלת קוד, אימות, השלמת הרשמה עובדת (fail-closed לא חוסם כי הסוד מוגדר).
- [ ] **עריכת עץ דורות** → admin/מזכירה עדיין יכולים לערוך (verifyStaff לא חוסם).
- [ ] **קביעת סיסמת פורטל חדשה** (admin) → נשמרת ומאפשרת כניסה (hash חדש).
- [ ] **טופס חלוקה חדשה** → אפשר להגיש גם בלי תקציב (הבאג תוקן).

## נדחה בכוונה (לא נכלל בפריסה זו)

1. **bucket הקבצים → private.** ה-URLs השמורים ב-DB הם public קבועים; הפיכה ל-private
   תשבור את כל המסמכים הקיימים עד רפקטור מלא של שכבת התצוגה (signed URLs בכל צפייה +
   migration של URLs קיימים). **פרויקט נפרד** — לא בטוח לעשות כ-migration בודד.

2. **IDOR ב-birth-request / loan-request.** הזרימה הנוכחית (ת"ז → lookup → הגשה) אין בה OTP.
   הוספת אימות דורשת **החלטת מוצר** + שינוי UI (שלב OTP לפני הגשה), אחרת שוברת את הגשת
   הבקשות הציבורית.

3. **הידוק RLS גרנולרי לפי role ספציפי** (admin מול secretary). הבסיס (`is_staff`) בטוח ולא
   שובר כלום; הידוק פר-role דורש מיפוי מדויק של הרשאות ונעשה בזהירות בהמשך.

4. **שלב 4 של migration הסיסמאות** (מחיקת עמודת `password` ה-plaintext) — הרץ רק אחרי
   כמה ימים של לוגין תקין מול ה-hash בפרודקשן.
